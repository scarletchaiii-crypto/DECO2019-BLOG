Last week's post showed what the interface looks like. This one is about
what has to exist in the database for any of that to work. The process
we followed goes wireframe → DDD → ERD.

## Annotating the wireframes

Before writing any tables, we went through each screen and labelled every
data element as D (displayed, read from somewhere), I (input, the user
provides it), or G (generated, the system produces it). The G fields are
the ones worth watching. Two of them showed up as columns in our first
instinct and had to be removed.

The "18 going" count on event cards is not stored anywhere. It is counted
from the `rsvp` table at query time. Storing it as a column on `event`
would mean keeping two things in sync every time someone RSVPs or cancels,
which is the kind of thing that goes wrong quietly and is annoying to debug.
The same applies to reply counts on forum posts.

## Writing the DDDs

## From DDDs to ERD

Once the entities were written out clearly, the relationships followed
from the data.

![Sitemap](assets/ERD.png)

`rsvp` is a junction table between `profile` and `event`, and it carries
a `status` attribute on the relationship itself. This is the same pattern
as the Pokémon ability example from the tutorial: whether an RSVP is
confirmed, attended, or cancelled is a property of the specific
user-event pairing, not of the user and not of the event independently.
Putting `status` on either of those tables would not make sense.

`comment` has two nullable foreign keys: `event_id` and `post_id`. On any
given row, exactly one of them holds a value. The alternative would be two
separate comment tables, one for event comments and one for post comments.
We decided against that because the comment structure is identical in both
cases, and maintaining two tables with the same shape creates more work
for no real benefit. The trade-off is that we need the application layer
to enforce the constraint that both cannot be null and both cannot have
values simultaneously.

`category` on both `event` and `post` is stored as a plain text field
rather than a foreign key to a separate categories table. The categories
are a fixed preset list with no additional attributes like colours or
icons in the current design, so a separate table would add joins without
adding anything useful. If the design changes later and categories need
their own properties, the schema would need updating, but for a prototype
that constraint is acceptable.

One thing `profile` does differently from the other tables is that
`user_id` is the primary key and has no auto-increment. The value comes
directly from the session cookie that the Bla+Bla platform provides. The
application creates a profile row the first time a user visits the
community, using the userId it already has from the session. There is no
separate sign-up flow.
